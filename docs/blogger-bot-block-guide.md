# Blogger 자동 발행 "봇 차단" — 원인 분석 & 재발 방지 가이드

> 작성: 2026-07-24 · 대상 블로그: **NEWS** (trend-cracker.blogspot.com, 2026-07-20 생성)
> 사건: API로 트렌드 기사를 자동 발행하던 중 Google이 계정을 **봇/남용으로 판정**해 발행을 차단.

---

## 0. 한 줄 요약 (TL;DR)

**신생 블로그(3일)에 API로 글을 수초 만에 몰아 올렸고(봇 시그니처), 차단된 뒤에도 12시간 넘게 100번+ 재시도하면서 "일시적 확인 요청"을 "지속적 봇 플래그"로 굳혀버린 것**이 원인이다.

- 우리가 부딪힌 건 **10,000/일 API 쿼터가 아니라**, ① 블로그당 하루 글 생성 상한(~50개)과 ② **계정 단위 봇 탐지("본인확인" SMS)** 였다.
- **결정타는 "차단된 POST를 계속 재시도한 것"** — 이게 봇 판정을 유지·악화시킨 가장 큰 실수다. `403`은 **재시도하면 안 되는 오류**다.
- 해결: **모든 자동 발행 즉시 중단 → 브라우저에서 수동 본인확인 → 며칠 동결 → 아주 천천히(하루 1~3개, 시간 간격 랜덤) 재개.**

## 1. 실제로 무슨 일이 있었나 (우리 사건 타임라인)

| 시각 | 행동 | 결과 |
|---|---|---|
| (초기) | **5편을 몇 초 만에** API 발행 | 즉시 `403 "Verify your info to continue"` (**SMS 본인확인** 요구) |
| ~09:00 | 소유자가 SMS 인증 완료 | 차단 해제 |
| 09:00~09:52 | **8편을 정확히 4분 간격**으로 발행 | 13편 도달 후 **다시 403 차단** (09:56) |
| 09:56~다음날 | 스크립트가 **4~20분 간격으로 100회+ 재시도** | **12시간+ 차단 지속, 안 풀림** → 계정 "봇" 플래그 고착 |
| 현재 | 13/28 발행, 지속 THROTTLE | 발행 불가 |

**핵심 패턴 3개가 전부 "사람이 아니다"라고 외쳤다:**
1. 수초에 5편 = 인간 불가능한 속도
2. 정확히 4분 고정 간격 = 기계(cron) 지문
3. 차단 후 12시간 기계적 재시도 = 봇이 벽을 두드리는 전형

## 2. 근본 원인 — 왜 "봇"으로 걸렸나

### 2-1. 헷갈리면 안 되는 **3개의 독립적 한도/시스템**

우리를 막은 건 서로 다른 3개다. 이걸 구분 못 하면 대응을 틀린다.

| # | 시스템 | 범위(scope) | 한도/트리거 | 해제 방법 |
|---|---|---|---|---|
| ① | **API 프로젝트 쿼터** | GCP 프로젝트/OAuth 클라이언트 | ~**10,000 요청/일**, ~100 요청/100초 | 자정 리셋 (우리는 여기 안 걸림) |
| ② | **블로그당 글 생성 상한** | **블로그 단위** | ~**50개/블로그/일**(스팸방지, 비공식) | ~24시간 후 리셋 |
| ③ | **계정 봇/남용 탐지** | **Google 계정 단위 (행동 기반)** | 버스트·기계적 간격·신생·서버IP | **사람이 브라우저에서 SMS 본인확인** |

> **우리가 실제로 걸린 건 ③(계정 봇 탐지).** 5편/수초는 ②의 50개 상한 근처도 아니었지만, **속도(velocity)** 만으로 ③이 즉시 발동했다. "10,000/일 쿼터 남았는데 왜 막혀?"의 답이 이것 — 쿼터는 우리 벽이 아니었다.

### 2-2. 우리가 밟은 지뢰 4개 (원인 순위)

1. **신생 블로그 + API 대량 발행 (신뢰도 0)** — 3일 된 blogspot은 신뢰가 거의 없어 남용 임계값이 가장 낮다. 오래된 블로그가 견디는 속도도 신생은 즉시 플래그된다.
2. **버스트(수초에 5편)** — 스팸 블로그(splog)의 1번 지문. 인간이 완결된 기사 5개를 10초에 못 올린다.
3. **고정 간격(정확히 4분)** — 볼륨보다 **더 강한 봇 신호**. 사람은 불규칙하게 올린다. cron의 시계추 같은 규칙성이 결정적 tell.
4. **⚠️ 차단 후 재시도 폭풍(12시간, 100회+)** — **가장 큰 실수.** `403`(PERMISSION_DENIED / rateLimitExceeded)은 **재시도 불가 오류**다. 재시도는 아무것도 못 풀고 남용 점수만 쌓는다. Google 남용 시스템은 임시 차단을 **분→시간→하루+로 계단식 악화**시키며, 재시도할수록 **일시적 확인 요청을 영구 플래그로 굳힌다.** (+ 서버/데이터센터 IP에서 100% API로만 활동 = 사람 흔적 0 = 스팸 팜처럼 보임)

### 2-3. (병렬) 콘텐츠 품질도 별개 트리거

속도와 무관하게, **얇거나·자동생성·중복** 콘텐츠를 신생·무트래픽·Search Console 미등록 블로그가 빠르게 쏟아내면 **splog(스팸블로그) 분류기**가 따로 잡는다. 즉 "천천히"만으로는 부족하고 **원본성·실질 내용**도 필요하다.

## 3. 지금 차단 푸는 법 (복구 절차 — 순서대로)

> ⚠️ **순서가 중요하다. Step 0을 먼저 안 하면 나머지가 무의미하다.**

0. **[필수·최우선] 모든 자동 발행 즉시 중단.** cron/스크립트/드립을 죽여서 **추가 POST가 계정에 0건** 닿게 한다. (완료함 — 드립 종료) 재시도 1건마다 플래그가 깊어진다.
1. **사람이 브라우저에서 본인확인 완료.** 소유 계정(kotsukotsu.days)으로 **일반 브라우저 + 집/폰 네트워크(주거용 IP)** 에서 로그인 → 대기 중인 "본인확인/SMS" 챌린지 완료. **서버/데이터센터에서 하지 말 것**(봇 신호 강화). `myaccount.google.com/security`에서 "계정 상태 양호" 확인.
2. **며칠 완전 동결.** 본인확인 후에도 **최소 24~48시간, 이상적으로 7일**간 API 발행 0건. 쿼터는 자정에 리셋되지만 **평판/신뢰 플래그는 시간 + 깨끗한 신호로만 감쇠**한다. (Google 민감작업 재확인 쿨다운이 7일)
3. **블로그가 LOCK 상태면 리뷰 요청.** 단순 throttle이 아니라 블로그가 "잠김/스팸 표시"면 Blogger 대시보드에서 **리뷰(이의제기) 요청** 제출 → 보통 **~영업일 2일** 내 검토. (미요청 시 ~20일 후 삭제, 재검토 옵션은 ~5일마다 재등장)
4. **사람 흔적 만들기.** API 재개 **전에** Blogger 웹 편집기로 **손으로 1~2편 발행/수정**. 100% 봇이 아니라는 히스토리를 남긴다.
5. **신뢰 신호 축적.** 블로그를 **Google Search Console에 등록**, About/연락처 페이지 추가, (가능하면) 실제 유입.
6. **OAuth 앱 정비.** 동의 화면을 **"프로덕션"으로 게시**(테스트 모드면 리프레시 토큰 7일 만료), 정비 후 **새 리프레시 토큰 재발급**.

## 4. 재발 방지 플레이북 (구체 수치)

### 4-1. 발행 속도 — **램프업 스케줄** (이걸 그대로 따르면 안 걸린다)

| 기간 | API 발행량 | 최소 간격 | 비고 |
|---|---|---|---|
| **1주차 (복구+워밍)** | **0편 (API)** + **수동 1편/일** | — | 사람 흔적만 |
| **2주차 (8~14일)** | **1편/일** | ~20시간 | 여기서부터 API 시작 |
| **3주차 (15~21일)** | **2편/일** | 4~6시간 | |
| **4주차 (22~28일)** | **3편/일** | 3~4시간 | |
| **2개월차 (29~60일)** | **4~6편/일** | 2~3시간 | |
| **3개월차+** | **8~10편/일** (자체상한 ~10~12) | 최소 45~90분 | 절대 ~50/일 근처 금지 |

**불변 규칙:**
- **두 글 사이 최소 45~90분**, 시간당 최대 1~2편.
- **발행 시간대: 07:00~23:00 (타깃 현지시간)** 만. 밤샘/24시간 금지.
- **간격 지터 ±25~50% + 랜덤 초 오프셋** → 정확한 cron 분(minute)에 절대 안 걸리게.
- **버스트 절대 금지** (수초·수분에 여러 편 ❌).

### 4-2. 재시도 규율 — **Circuit Breaker (가장 중요)**

- **`403`(PERMISSION_DENIED / "verify your info" / 남용)은 FATAL = 재시도 0회.** 전체 파이프라인 중단 + **사람에게 알림**. (API 경로로는 챌린지를 구조적으로 못 푼다.)
- 재시도는 **`429`/`500`/`503`만**, **캡드 지수 백오프 + 지터**: `wait = min(2^n + rand(≤1000ms), 32~64s)`, **최대 3~5회 후 그날은 포기**.
- **서킷 브레이커**: `403` **1회 → 그날 발행 전면 정지**; **2회 → 시스템 정지 + 사람 리뷰 대기**.
- `429`(속도)·`403 quotaExceeded`(일일캡)·`403 PERMISSION_DENIED`(본인확인) 를 **구분해서** 대응. `Retry-After` 헤더 존중.
- **모니터링**: 매 API 응답 로깅, **첫 403에서 즉시 알림**(12시간·100회 뒤에 발견하지 말 것).
- **멱등성**: 재시도가 같은 글을 **재발행하지 않게**(중복/유사도 플래그 방지) — 우리는 `.publish-state.json` 해시로 이미 방지 중.

### 4-3. 계정/블로그 신뢰 쌓기

- 블로그 **나이**(오래될수록 임계값↑), About/프로필/연락처 페이지, **Search Console 등록**, 실제 구독/유입, (가능하면) 커스텀 도메인.
- **100% API 금지** — 가끔 웹 편집기로 손수 발행/수정해서 "사람+봇 혼합" 패턴 유지.

### 4-4. 콘텐츠 품질 (속도와 별개)

- 모든 글 **원본·실질·서로 다르게**(스핀/중복/얇은 글 ❌). 저(低)엔트로피 콘텐츠는 스팸 신호.
- 외부/제휴 링크는 과하지 않게.

### 4-5. 절대 하지 말 것 (Don'ts)

- ❌ 차단된 POST **재시도** (이번 사건의 주범)
- ❌ 본인확인을 **서버/데이터센터 IP**에서 시도 (봇 신호 강화)
- ❌ **버스트** (수초·수분에 여러 편)
- ❌ **고정 간격**(정확한 4분 같은 cron 규칙성)
- ❌ 신생 블로그에 **대량 볼륨**
- ❌ 차단 회피용으로 **새 블로그/새 계정 생성** (회피는 탐지되고 **계정 단위 제재로 격상**될 수 있음)
- ❌ "혹시 풀릴까" **cron 계속 돌리기** (쿼터는 재시도와 무관하게 자정 리셋, 본인확인 플래그는 재시도로 절대 안 풀림)
- ❌ **10,000/일 쿼터를 상한으로 착각** (블로그 50/일 + 계정 봇플래그가 훨씬 먼저 막음)
- ❌ ~50~60/일 **근처 접근** (그 구간이 CAPTCHA/확인 발동선)
- ❌ 얇은/자동생성 **중복 콘텐츠 대량 발행**

## 5. 우리 코드/운영에 적용할 구체 변경

현재 `scripts/lib/publisher.mjs`·`blogger.mjs`·`drip-publish.sh`를 이렇게 바꿔야 한다:

1. **`blogger.mjs`**: 현재 403(rateLimitExceeded)도 백오프 재시도함 → **모든 403은 재시도 0회로 즉시 throw**. 429/5xx만 캡드 백오프(최대 3~5) + 지터.
2. **`publisher.mjs`**: **일일 발행 하드캡**(예: 초기 1~3, 환경변수 `MAX_POSTS_PER_DAY`)과 **글 간 최소 간격(45~90분) + 지터** 내장. 캡 도달 시 그날 종료.
3. **서킷 브레이커**: 403 발생 → 상태파일에 `blockedUntil`(예: +24h) 기록하고 그 전까지 발행 스킵. 403 2회 누적 → 완전 정지 + 알림.
4. **`drip-publish.sh` 폐기 또는 대체**: 고정 간격·무한 재시도 방식은 위험. 대신 **하루 1~3회, 랜덤 시각(07~23시)** 에 도는 스케줄(launchd/cron)로, **한 번에 1편만**, 403이면 즉시 종료.
5. **알림**: 첫 403에서 즉시 사람에게 통지(로그/텔레그램 등).
6. **모니터링**: 발행 로그에 응답코드·시각 기록, 이상 시 조기 경보.

> 원하면 위 1~5를 실제 코드로 반영해줄 수 있음. (이번 사건의 재발을 코드 차원에서 막는 핵심은 **"403 = 즉시 중단, 재시도 금지" + "하루 소량 + 랜덤 간격"** 둘이다.)

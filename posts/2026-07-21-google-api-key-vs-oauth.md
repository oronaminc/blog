Google API를 처음 쓰면 이런 의문이 든다. "어떤 건 API 키만 넣으면 되는데, 어떤 건 왜 OAuth라는 복잡한 절차를 거쳐야 하지?" 이 글에서 그 차이를 정리한다.

## 한 줄 요약

- **API 키** = "이 **앱**이 누구인지" 증명 → **공개 데이터** 접근
- **OAuth 2.0** = "이 **사용자**가 허락했는지" 증명 → **특정 사용자의 개인 데이터·수정**

## API 키

API 키는 프로젝트(앱)를 식별하는 문자열이다. 요청에 키를 붙이면 구글은 "○○ 프로젝트가 호출했군" 하고 사용량을 집계하고 할당량을 적용한다.

**쓰는 곳** — 누구의 것도 아닌 공개 데이터/서비스. 지도 타일, 번역, 공개 검색 등.

**한계** — API 키는 "누가 허락했는지"를 담지 못한다. 그래서 개인 계정의 **비공개 데이터에 접근하거나 수정**할 수 없다.

## OAuth 2.0

OAuth는 "사용자가 이 앱에게 특정 권한을 위임했다"는 것을 증명하는 절차다. 사용자가 동의 화면에서 "허용"을 누르면, 앱은 그 사용자를 대신해 정해진 범위(scope) 안에서 행동할 수 있는 토큰을 받는다.

핵심 개념:

- **동의 화면(Consent screen)** — 사용자가 어떤 권한을 주는지 확인하고 허용
- **범위(Scope)** — "블로그 관리", "메일 읽기"처럼 권한의 경계
- **액세스 토큰** — 보통 1시간짜리 단기 토큰
- **리프레시 토큰** — 액세스 토큰을 계속 재발급받는 장기 토큰 (자동화에 필수)

**쓰는 곳** — Gmail, Drive, Calendar, **Blogger**, YouTube 업로드 등 개인 계정의 데이터를 읽거나 쓰는 모든 것.

## 표로 비교

| | API 키 | OAuth 2.0 |
|---|---|---|
| 증명 대상 | 앱 | 사용자의 동의 |
| 접근 범위 | 공개 데이터 | 개인 데이터 + 쓰기 |
| 사용자 로그인 | 불필요 | 필요 (최초 1회) |
| 대표 예시 | 지도·번역 | Gmail·Blogger |

## Blogger의 경우

블로그 글을 읽기만 하는 공개 조회는 API 키로도 된다. 하지만 **글을 발행·수정**하는 건 "내 블로그에 글을 쓴다" = 특정 계정의 자산을 바꾸는 행위라, 반드시 OAuth가 필요하다. 실제로 소유자가 아닌 계정의 토큰으로 발행을 시도하면 `403 Forbidden`이 뜬다.

## 자동화에서 흔한 함정 2가지

1. **동의 화면이 "테스트" 상태면 리프레시 토큰이 7일 만에 만료된다.** 앱을 "프로덕션"으로 게시해야 장기 토큰을 받는다.
2. **리프레시 토큰은 최초 인증 때 한 번만 발급된다.** 인증 URL에 `access_type=offline`과 `prompt=consent`가 있어야 하고, 이미 권한을 준 앱은 재발급이 안 될 수 있다.

## 정리

> 남의(=내) 계정을 **대신 조작**하는 API라면 OAuth, 그냥 **공개 서비스를 소비**하는 거라면 API 키.

마크다운으로 글을 쓰고, 브라우저에서 관리하고, `git push` 한 번으로 Blogger에 자동 발행되는 시스템을 만들었다. 이 글은 그 구조와 개발 과정을 정리한 것이다. (이 글 역시 그 시스템으로 발행됐다.)

## 무엇을 만들었나

두 층으로 되어 있다.

1. **자동발행 파이프라인** — `posts/`의 마크다운을 Blogger API로 발행. `git push`하면 GitHub Actions가 실행.
2. **로컬 관리 CMS** — 브라우저(`localhost`)에서 글을 쓰고, 상태를 보고, 클릭으로 발행하는 대시보드.

## 파이프라인 설계

- 글은 `posts/*.md` (제목·라벨·날짜·초안 여부는 상단 frontmatter)
- 마크다운 → HTML 변환 후 Blogger REST API v3로 전송
- `.publish-state.json`에 `파일 → {postId, 내용해시}`를 기록
  - **새 글**은 생성, **바뀐 글**은 업데이트, **그대로면 스킵** → 중복 발행 없음
- 인증은 OAuth 리프레시 토큰 (API 키가 아니라 OAuth를 쓰는 이유는 별도 글에서 정리했다)

의존성을 최소화하려고 무거운 SDK 없이 `fetch`로 REST를 직접 호출했다.

## 로컬 CMS

`git push`만으로도 발행되지만, 글을 편하게 쓰고 관리하려면 UI가 필요했다. 그래서 Express + 빌드 과정 없는 바닐라 JS로 로컬 대시보드를 붙였다.

- **글 목록 + 상태 배지** — 로컬만 / 초안 / 발행됨 / 수정됨(재발행 필요)
- **에디터 + 실시간 프리뷰** — 발행될 HTML과 똑같이 렌더
- **원클릭 발행 / 초안 ↔ 공개 전환**
- **검색·필터**, **원격 글 가져오기(임포트)**

### 설계에서 가장 신경 쓴 것: "하나의 진실"

배지가 보여주는 상태와 실제 발행 결과가 어긋나면 최악이다. 그래서 **해시 계산·발행 로직을 단 하나의 모듈**로 만들고, CLI·웹서버·GitHub Actions가 전부 그걸 공유하게 했다. 프리뷰도 발행에 쓰는 렌더러를 그대로 사용한다. "두 번째 구현을 만들지 않는다"가 규칙.

## 삽질과 교훈

- **OAuth 계정** — 블로그 소유 계정으로 인증해야 한다 (다른 계정이면 403).
- **7일 만료** — 동의 화면을 "프로덕션"으로 게시해야 리프레시 토큰이 오래 산다.
- **이미지** — Blogger API는 이미지를 호스팅하지 않는다. 절대 URL로 넣어야 한다.
- **보안** — 파일 쓰기·발행 권한이 있는 로컬 서버는 `127.0.0.1`로만 열었다.

## 설계는 어떻게 검증했나

혼자 감으로 만들지 않고, 기존 git 기반 CMS들(Decap, Sveltia, Tina, Pages CMS, Keystatic)의 구조를 조사해 공통 패턴을 뽑아 반영했다. 결론적으로 이런 도구들의 코어는 거의 동일했다 — **상태 목록 → 폼 기반 프론트매터 에디터 + 라이브 프리뷰 → 원클릭 저장·발행**.

## 앞으로

지금도 충분히 쓸 만하지만, 이미지 드래그 업로드·라벨 자동완성·일괄 발행·원격 변경 감지 같은 편의 기능을 더 붙일 수 있다.
